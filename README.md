# 历史印记 Heritage Map

> 一个面向人文历史爱好者的地图产品。用户不仅可以“看地图”，还可以围绕古建筑、历史遗址、纪念地与城市记忆，进行打卡、巡礼、评论、收藏与路线规划。

## 项目简介

**历史印记** 是一个人文向地图产品 Demo，聚焦“历史空间 + 内容叙事 + 个人探索”三件事：

- 用地图承载古建筑、历史遗址、纪念地等 **线下空间**
- 用结构化信息与创作者说明补足地点背后的 **历史语境**
- 用收藏、评论、路线规划与自建地图，把“看见地点”升级为“完成一次主题巡礼”

它不是传统旅游地图，也不是单纯的 POI 聚合工具，而是希望服务这样一类用户：

> **他们不是只想知道“哪里值得去”，而是想知道“为什么值得去、应该按什么主题去、去了之后能获得怎样的文化体验”。**

在当前版本中，产品已经支持：公开地图浏览、地图详情查看、地点展开、地图/地点评论、地图收藏、路线规划、登录注册、三步式地图创建，以及基于维基百科摘要与 POI 搜索的辅助创作流程。

## 产品定位

### 目标用户

- 喜欢古建筑、近代史、城市史、博物馆和遗址空间的人文历史爱好者
- 会专门做“圣地巡礼”“主题 walk”“城市微旅行”的深度探索型用户
- 想把自己的研究、兴趣路线或城市观察整理成专题地图的内容创作者

### 核心需求

这类用户通常有三个连续需求：

1. **发现**：找到值得去的历史地点与专题路线
2. **理解**：快速了解地点背后的历史、人文与建筑信息
3. **行动**：真的把这些地点串联起来，完成一次可执行的线下巡礼

**历史印记** 的产品设计，就是围绕这条链路展开的。

## 我想解决的问题

市面上很多地图产品更擅长回答“怎么去”，却不擅长回答：

- 这些地点为什么应该被放在一起看？
- 一个地点除了地址，还有哪些值得被记录的历史信息？
- 如果我想按一个主题去走，比如“抗战旧址”“民国建筑”“城市记忆”，有没有现成的地图与顺序？

因此，这个项目希望把地图从“工具型导航”升级成“主题型探索产品”：

- **地图不只是底图，而是内容容器**
- **地点不只是坐标，而是故事入口**
- **路线不只是路径，而是一次主题巡礼的组织方式**

## 核心功能

### 1. 地图广场：先看主题，再决定去哪里

首页展示所有公开地图，采用卡片化方式呈现：

- 地图标题、主题、简介
- 地点数量、浏览量、收藏量
- 创作者信息
- 官方示范地图兜底展示

这意味着产品不是从“搜一个点”开始，而是从“选一个主题”开始，更符合人文探索类产品的使用方式。

### 2. 地图详情：把“地点列表 + 地图视图”组织成一个完整阅读界面

进入地图详情后，用户会同时看到：

- 地图视图中的编号 Marker
- 右侧地点列表
- 地点展开后的历史介绍、建筑信息、开放信息
- 地图级评论区
- 收藏入口
- 路线规划入口

这种设计兼顾了两件事：

- **空间感**：地点之间的地理关系是可视化的
- **叙事感**：地点不是孤立 POI，而是可连续阅读的内容单元

### 3. 地点信息结构化：让地图具备“策展”属性

每个地点不仅有名称、地址、坐标，还支持补充：

- 历史/人文介绍
- 建筑风格
- 建筑年代
- 设计/建造者
- 开放时间
- 票价信息
- 创作者个性说明

这让地图创建者不只是“标点”，而是在做一张有主题立场和内容密度的策展式地图。

### 4. 路线规划：把“想去”变成“能去”

在地图详情页中，用户可以：

- 勾选本次想游览的地点
- 指定起点与终点
- 使用预设地点或自定义地址
- 按“从近到远 / 从远到近”生成游览顺序
- 查看驾车、步行、公交、骑行等多种路线方案
- 在地图上显示路线

这个功能的价值不只是导航，而是把专题地图真正转化为一次可执行的线下行动。

### 5. 互动机制：让地图从静态内容变成社区内容

当前版本已支持：

- 用户注册 / 登录
- 地图收藏
- 地图级评论 + 评分
- 地点级评论

这意味着产品开始具备社区雏形：

- 地图可以沉淀公共价值
- 地点可以沉淀个体体验
- 收藏行为可以帮助识别更有吸引力的主题内容

### 6. 创作工作流：降低“做一张地图”的门槛

创建地图采用 **三步式向导**：

1. 填写地图基本信息
2. 添加地点
3. 预览并发布

在添加地点时，又提供了两类辅助能力：

- **POI 搜索**：根据地点名称候选匹配坐标
- **地址地理编码**：根据详细地址自动解析经纬度
- **维基百科摘要辅助**：帮助创作者快速生成地点的人文介绍初稿

这使得“做地图”不只是专业用户的能力，也成为普通创作者可以完成的内容生产行为。

### 7. 新手引导：降低首次使用成本

产品内置了 onboarding tour，用分步骤方式向首次访问用户解释：

- 如何浏览地图
- 如何查看地点详情
- 如何评论与收藏
- 如何规划路线
- 如何创建自己的地图

对于一个功能链路较长的新产品，这类引导能显著降低理解成本。

## 用户体验设计思路

### 为什么采用“深色 + 金红色”的视觉语言？

这个产品不是工具型出行 App，而更接近“历史内容 + 巡礼体验”的复合场景。

因此视觉上刻意避免了常见生活服务类地图的轻快风格，而是选择：

- 深色背景：强化沉浸感与仪式感
- 金色点缀：对应历史、纪念、典藏感
- 红色强调：用于路径、按钮与关键交互，增强行动性

这套语言希望传达的是：

> 你不是在刷一个地点列表，而是在进入一段需要被认真对待的历史路径。

### 为什么把地点信息做成“展开式侧栏”？

因为这类产品的核心不是“快速切页”，而是“在一个主题里连续阅读多个地点”。

展开式侧栏有三个优点：

- 避免频繁跳转，阅读更连贯
- 与地图联动，空间与内容同时保留
- 更适合做主题巡礼中的连续比较

## 产品亮点

从产品经理视角，我认为这个 Demo 最值得展示的不是“做了一个地图页面”，而是以下几点：

### 1. 从工具逻辑转向主题逻辑

传统地图产品关注“查一个点”，而这个产品更关注“围绕一个主题串起一组点”。

### 2. 从信息展示转向体验组织

不仅展示地点，还把评论、收藏、路线规划、地图创建串成一条完整链路，让用户可以从浏览进入行动。

### 3. 兼顾消费侧与供给侧

- 对浏览者：可以发现、理解、规划、评论、收藏
- 对创作者：可以低门槛创建自己的专题地图

这意味着它不是单纯的内容页，而是具备 UGC 扩展能力的产品雏形。

### 4. 有明确的人文产品气质

无论是文案、配色、信息结构还是示范地图选题，产品都在尽量建立一种统一的品牌感：

- 不是“去哪儿”
- 而是“为什么去、怎么以一个主题去”

## 技术实现

### 前端

- HTML / CSS / JavaScript 原生实现
- 响应式布局，兼容桌面端与移动端基础浏览

### 地图与地理能力

- **Leaflet + OpenStreetMap**：负责底图展示与 Marker 渲染
- **高德地图 JS API**：负责 POI 搜索、地址解析、驾车 / 步行 / 公交 / 骑行路线规划

### 后端 / 数据能力

- **Supabase Auth**：用户注册、登录、登出、认证状态管理
- **Supabase Database**：地图、地点、评论、收藏等数据存储
- **Supabase Storage**：预留图片上传能力

## 我作为产品经理的思考

这个项目对我来说，不只是“实现了一个地图产品 Demo”，更重要的是验证了一个产品方向：

> **地图可以不仅服务效率，也服务记忆、理解与文化体验。**

很多线下空间本身并不会“自动讲故事”，真正让用户感受到它们意义的，是被组织过的主题、信息与路径。

而“历史印记”想做的，就是把这些分散的人文地点，组织成一张真正值得被走一遍的地图。
